The AI Tech Lead Framework

Governed, consistent, measurable AI-assisted development.

One source of truth that every AI tool reads and obeys — turning our conventions into something the AI can't ignore.

.NET & Angular · Runs entirely on local Bitbucket + Windows · No cloud dependency

Opening line: "AI is already writing a meaningful share of our code. This is how we make sure it writes our code, to our standards — and how we prove it."

Set the frame: this is not a new tool to learn so much as a set of rails that make our existing standards automatic. Two audiences today: leads first (governance + impact), then teams (daily workflow).

Keep this slide up while people settle. 30 seconds.

The Shift

AI now writes a growing share of our code

  • Copilot and Claude are in every IDE; delivery speed is genuinely up.
  • But the output is ungoverned — it varies by developer, by tool, and by prompt.
  • The assistant doesn't know our conventions, our architecture, or our "never do this" list.
AI accelerates whatever you point it at — including our mistakes.

Goal: establish the tension. Speed is real and good; the lack of governance is the risk.

Concrete example to say aloud: "Three developers ask Copilot for 'a service to do X' and get three different shapes — different error handling, different layering, different testing. Multiply that across a year."

Don't dwell on doom — pivot fast to "there's a clean way to fix this."

The Problem

Speed without guardrails

Invents APIs

Confidently calls methods, packages, and endpoints that don't exist in our codebase.

Adds bloat

Premature abstractions, speculative "future-proofing," wrappers that do nothing.

Hides failures

Wraps errors in try/catch, silences analyzers with #pragma to make red go green.

Ignores conventions

Doesn't know our layering, naming, DI, or testing patterns unless told every time.

Inconsistent output

Every dev and every tool produces a different shape for the same task.

No audit trail

No record of what the AI changed, or why — a governance and review blind spot.

Goal: name the failure modes precisely so leads recognize them from their own repos.

Ask the room: "How many of these have you seen in a PR this month?" — gets nods, builds buy-in.

These six map directly to the framework's four pillars (next slides). Each problem has a countermeasure.

The Idea

An operating system for AI-assisted development

  • A repo-level standard every AI tool reads and obeys — authored once.
  • Tool-agnostic. Claude Code, GitHub Copilot (agent & CLI), Cursor, Gemini, Aider — same rules.
  • Adopted as a template — drop it into a repo; it learns your codebase.
  • Two stacks in lockstep — a .NET edition and an Angular edition, kept identical in spirit.
It doesn't replace the AI. It governs it.

Key reframe: not "another AI tool" — a layer that sits over the tools we already use. People relax when they hear they don't have to switch assistants.

"Authored once" is the headline — say it twice. The whole value chain hangs off a single file.

  • Generated, not hand-copied — mirrors are produced from the source file.
  • Drift-checked in CI — the build fails if a mirror falls out of sync.

Goal: show the mechanism that makes "authored once" true. Edit CLAUDE.md → regenerate → all tools updated.

Why mirrors at all? Because each tool reads a different file. We don't maintain four sets of rules — we maintain one and project it.

The drift check is what stops this rotting in six months — emphasize for leads who've seen docs go stale.

The Non-Negotiables

Four pillars

Verification

Verify before you reference. No invented APIs, types, or packages — read the source or say "unverified."

Leanness

No bloat, no premature abstraction, no future-proofing. Deletion is a contribution.

SOLID

Mandatory and literal — every injected service behind an interface, with a deterministic CI backstop.

Boy Scout Rule

Leave every touched file cleaner — symmetric: add what's missing, remove dead weight.

Goal: these are the standards, encoded. They're the same ones a good tech lead would enforce in review — now they're enforced continuously.

SOLID note: the tech lead asked for literal SOLID. Verification + Leanness are reconciled (no bloat, but service seams are required). Mention the CI backstop is deterministic — not just an LLM opinion.

If asked "isn't leanness vs SOLID a contradiction?" — no: SOLID governs structure, leanness governs ceremony beyond that structure.

Teeth

Enforced — not just documented

⛔ Real-time hooks

A pre-write guard blocks suppressions and secrets before they hit disk. Every AI edit triggers a build + an audit-log entry.

🔍 Review subagents

Specialised reviewers for security, SOLID, conventions, and bloat run on demand — a second pair of eyes that never gets tired.

🧱 CI guardrails

Architecture tests (dependency direction) and a docs-sync check fail the pipeline on violations or drift.

🧾 Audit trail

Every AI-assisted change is appended to an audit log; security findings tracked separately with remediation SLAs.

Rules that aren't enforced are just suggestions.

This is the slide leads care about most. The difference between a style guide nobody reads and a system that holds the line.

Concrete: the PreToolUse hook hard-blocks #pragma warning disable and obvious secrets — the AI literally cannot write them.

Bitbucket note: CI guardrails run via Bamboo/Jenkins or a pre-receive hook, and publish to PRs through Bitbucket Code Insights. No cloud needed.

Practical Implications — Tech Leads

What changes for you

You gain

  • Your conventions become executable, not tribal knowledge.
  • Consistency across the team and across AI tools.
  • Governance: an audit trail and tracked security findings with SLAs.
  • You spend review time on design, not catching boilerplate slips.

You own

  • Keeping CLAUDE.md current — it's now a first-class artifact.
  • Deciding conventions deliberately (they're code now, so they carry weight).
  • Championing adoption: one repo, then the team.

Goal: make the lead see themselves in it. The framework amplifies them — their standards, applied at machine scale and consistency.

"You own CLAUDE.md" is the honest cost. It's small (a few hundred lines, budgeted) and the drift checks help keep it honest, but it is a living document.

Land the review-time point hard: less time on "you forgot a CancellationToken," more on "is this the right design."

Practical Implications — Developers

What changes for you

Easier

  • Slash commands run the whole workflow — /feature, /fix, /review, /design.
  • Skills auto-trigger for common tasks (add endpoint, add entity, register service).
  • The AI already knows our conventions and structure — less prompting.
  • Guardrails catch slips before code review, so PRs go smoother.

Different

  • You state intent; the framework drives the steps.
  • No silencing failures — fix the cause, not the symptom.
  • SOLID & leanness are enforced — yes, even on a deadline.

This is the team slide. Lead with what makes their day easier, then the constraints — framed as "the rails that keep us out of the ditch."

Demo idea for the team session: run /feature on a small change live and let them see the plan → verified subtasks → self-review.

Pre-empt the pushback: "enforced on a deadline?" Yes — because the friction is cheaper than the cleanup. But hooks have an escape hatch for genuine hotfixes (documented).

Proof, Not Faith

We measure the difference

Adopting the framework runs a before/after automatically — no manual setup.

Tier 1

Capability diff

What the old setup lacked vs what's now in place, gap by gap.

Tier 1

Deterministic scorecard

Anti-patterns, SOLID/DIP, security, test counts — measured, baselined at adoption.

Tier 2

Behavioral A/B

The same tasks run through a real agent against the old framework vs the new one, several trials each.

Same model, same tasks — the only difference is the framework. Honest, repeatable evidence.

Differentiator. Most "AI standards" decks ask for faith. This produces a report on your codebase.

Be honest about stochasticity: LLM runs vary, so we run trials and read distributions, not a single number. The capability diff is concrete immediately; the A/B is the behavioral evidence.

Before the meeting: run /adopt on a pilot repo and paste the real headline numbers here. Don't invent figures — the whole point is that they're measured.

No Cloud Required

Built for our environment

Local & self-hosted

VS Code + Copilot, Claude Code, and the Copilot CLI all run locally. Works with Bitbucket Data Center — nothing leaves the building it shouldn't.

CI on our terms

Guardrails run in Bamboo / Jenkins / a pre-receive hook and report to PRs via Bitbucket Code Insights.

Windows-first

Every script ships as a bash and a PowerShell twin.

Both stacks

.NET (NetArchTest, EF Core) and Angular (dependency-cruiser, RxJS) — same doctrine, stack-appropriate teeth.

Goal: remove the "but we're on local Bitbucket, not GitHub cloud" objection before it's raised.

Explicitly: GitHub Actions / Copilot cloud agent / Rovo Dev are not required and not assumed. The framework is host-agnostic; CI is wired to whatever we already run.

The PowerShell-twin point matters here — this was built knowing the team is on Windows.

Straight Talk

Honest tradeoffs

Token cost

More context per turn. Mitigated by model routing, slim instructions, and prompt-cache-stable files.

Discipline cost

SOLID and leanness are enforced. The friction is the point — but it is friction.

Learning curve

A handful of slash commands and a new habit: edit CLAUDE.md, don't fight it.

It isn't magic. It makes good practice the path of least resistance.

Credibility move. Naming the costs yourself earns trust and disarms skeptics. Leads have seen enough silver-bullet pitches.

Token cost is real but actively managed — cheaper agents for cheap checks, the heavy model only where it matters, and a tight per-turn base file.

If pressed on ROI: the cost is tokens + discipline; the return is consistency, fewer review cycles, and a measurable quality delta.
